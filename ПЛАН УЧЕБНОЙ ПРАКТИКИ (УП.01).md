#**ПЛАН УЧЕБНОЙ ПРАКТИКИ (УП.01)**
для обучающихся по 09.02.07 «Информационные системы и программирование» 2 курс, 4 семестр (в рамках освоения профессионального модуля ПМ.01 «Разработка программных модулей») специальности

####1. Вид практики: Учебная практика.
####2. Способ проведения: Рассредоточенная, на базе научной лаборатории колледжа.
####3. Основание: Рабочая программа профессионального модуля ПМ.01 и учебный план специальности.

####4. Цели практики:
*   Сформировать первоначальный практический опыт разработки полнофункционального веб-приложения.
*   Закрепить навыки работы с современными стеком-технологиями: Python (FastAPI), PostgreSQL, Nest.JS/React и Telegram Bot API.
*   Развить общие и профессиональные навыки в области задач анализа, проектирования, программирования, тестирования и документирования.

####5. Задачи практики:
*   Проанализировать техническое задание по разработке информационных систем.
*   Спроектировать архитектуру приложения «клиент-сервер-база данных».
*   Реализовать backend-API на Python с использованием FastAPI и интеграцией с PostgreSQL.
*   Разработать клиентское приложение на Nest.JS или React либо Telegram-боте для взаимодействия с API.
*   Протестировать взаимодействие всех компонентов системы.
*   Оформить проектную документацию и исходный код в Git-репозитории.
*   Подготовить отчет и защитить результаты работы.

####6. Место и время пребывания:
Место: Учебные информационные технологии колледжа.
Срок: 4 недели в соответствии с календарным учебным графиком.
Трудоемкость: 144 часа (36 часов в неделю).

####7. Тема итогового проекта:
Разработка прототипа информационной системы «Система измерения работ» с использованием технологий Python (FastAPI), PostgreSQL, Nest.JS/React и Telegram Bot.

####8. Содержание работы и календарный график:

#####Неделя 1. Проектирование и подготовка специалистов.
*   Содержание работы:
    1.  Анализ технического задания. Определение требований к системе.
    2.    Разработка компонентов схем взаимодействия (API, БД, Frontend, Bot).
    3.    Проектирование логической модели базы данных. Создание ER-диаграмм.
    4.    Настройка рабочего окружения: установка и настройка IDE VSCode, СУБД PostgreSQL, Python, Node.js.
    5.    Создание и настройка Git-репозитория на GitHub. Название начального .gitignore.
*   Контрольный результат недели:
     *   Утвержденная схема конструкции приложения.
     *   ER-диаграмма базы данных и SQL-скрипт для создания таблиц (init_db.sql).
     *   Созданный и доступный репозиторий на GitHub.

#####Неделя 2. Разработка серверной части (Backend API).
*   Содержание работы:
    1.  Инициализация FastAPI-приложения. Настройка структуры проекта.
    2.  Подключение к базе данных PostgreSQL с использованием асинхронных драйверов (asyncpg) и ORM (SQLAlchemy).
    3.  Создание Pydantic-моделей для валидации данных и SQLAlchemy-моделей для работы с БД.
    4.  Разработка CRUD-эндпоинтов для основных оснований «Задача» (создание, чтение, обновление, удаление).
    5.  Добавление условий маршрутизации и обработки ошибок.
    6.  Настройка CORS для обеспечения взаимодействия с будущим интерфейсом.
*   Контрольный результат недели:
    *   Рабочее FastAPI-приложение, запускаемое локально.
    *   Сбор проверенных запросов к API в Postman или Insomnia.
    *   Реализованные основные конечные точки, работающие с тестовой базой данных.

#####Неделя 3. Разработка клиентской части (Frontend/Telegram Bot).
*   Содержание работы (по выбору студента или по указанию руководителя):
    *   Вариант А (Веб-интерфейс на Nest.JS/React):
        1.  Создание проекта на выбранном фреймворке.
        2.  Разработка пользовательского интерфейса для отображения задач таблицы и формы создания/редактирования.3. Реализация логики для отправки HTTP-запросов (fetch/axios) к разработанному API.
        3.  Изображение отображения данных, полученных с сервера.
    *   Вариант Б (Telegram Бот):
        1.  Создание нового бота через @id199142634 (@BotFather) и получение токена.
        2.  Разработка бота на айограмме библиотеки (асинхронной).
        3.  Реализация команд (например, /start, /tasks, /new) и обработка обратных запросов с кнопок.
        4.  Интеграция бота с ранее созданным API: отправка запросов для получения списка задач, создание новой задачи.
*   Контрольный результат недели:
    *   Рабочее клиентское приложение (локальный веб-сервер или функционирующий Telegram-бот).
    *   Демонстрация полного цикла: действие в клиенте -> запрос к API -> изменение в БД -> ответ клиенту.

#####Неделя 4. Интеграция, обучение, защита.
*   Содержание работы:
    1.  Комплексная проверка работы всей системы. Сквозное обсуждение.
    2.  Отладка и исправление выявленных ошибок. Рефакторинг кода.
    3.  Написание ключевых модульных тестов для API (с использованием pytest).
    4.  Оформление проекта: создание подробного README.md с описанием, инструкцией по установке и запуску.
    5.  Подготовка итоговой презентации (5-7 слайдов) и демонстрационного сценария.
    6.  Написание и оформление отчета по замене.
    7.  Защита проекта перед аттестационной комиссией.
*   Контрольный результат недели:
    *   Стабильно работающий прототип информационной системы.
    *   Заполненный Git-репозиторий с качественным кодом и документацией.
    *   Завершенный письменный отчет по ссылке.
    *   Презентация и успешная защита.

####9. Формы Великобритании:
1.  Отчет по учебной схеме в печатном виде содержит все структурные элементы.
2.  Программный продукт: ссылка на публичный Git-репозиторий (GitHub/GitLab) с исходным кодом и настоящими коммитами.
3.  Презентация для защиты в формате PPTX или PDF.
4.  Устный доклад и демонстрация работы приложения на защите.

####10. Критерии оценки результатов:
*   Исправление реализации проекта (40%): функциональная полнота, работоспособность, корректность компонентов.
*   Техническое качество кода (30%): чистота, структурированность, комментарии, логичность истории Git.
*   Качество документации и отчета (20%): полнота, грамотность, четкость описания.
*   Защита проекта (10%): качество доклада, результаты, ответы на вопросы.
