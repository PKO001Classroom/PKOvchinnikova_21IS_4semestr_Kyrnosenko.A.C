## 1. Обзор системы

### 1.1. Назначение системы
Система "EduGrader" предназначена для автоматизации процесса оценивания учебных работ в образовательных учреждениях. Система предоставляет инструменты для создания заданий, загрузки работ, их проверки и анализа результатов.

### 1.2. Основные функции
- Управление пользователями и доступом
- Создание и управление учебными курсами
- Публикация заданий с критериями оценки
- Загрузка и хранение студенческих работ
- Проверка работ преподавателями
- Формирование отчетов и аналитики

## 2. Акторы системы

### 2.1. Основные акторы

| Актор | Описание | Пример использования |
|--------|------------|----------------------|
| **Студент** | Учащийся, выполняющий учебные задания | Загрузка лабораторной работы, просмотр оценок |
| **Преподаватель** | Педагог, проверяющий работы | Создание задания, проверка работ, анализ успеваемости |
| **Администратор** | Управляющий системой | Управление пользователями, настройка системы |
| **Гость** | Неаутентифицированный пользователь | Регистрация в системе, просмотр публичной информации |

### 2.2. Характеристики акторов

**Студент:**
- Частота использования: ежедневно в учебный период
- Техническая подготовка: средняя/высокая
- Основные потребности: удобная загрузка работ, понятная система оценивания

**Преподаватель:**
- Частота использования: 3-8 раз в неделю
- Техническая подготовка: варьируется
- Основные потребности: экономия времени при проверке, аналитика успеваемости

**Администратор:**
- Частота использования: 1-3 раза в неделю
- Техническая подготовка: высокая
- Основные потребности: управление доступом, мониторинг системы

### 2.3. Внешние системы (External Actors)
- Почтовый сервис/SMTP — для отправки уведомлений.
- Файловое хранилище (например, S3) — для хранения загружаемых работ.
- Сервис аутентификации вуза (задел на будущее, например, LDAP/Active Directory) — для единого входа (SSO).

## 4. Детальное описание use cases

### UC-01: Регистрация в системе

**Описание:** Создание новой учетной записи пользователя

**Основной поток:**
1. Гость переходит на страницу регистрации
2. Заполняет форму: email или телефон, пароль, подтверждение пароля, имя, фамилия, отчество, роль, группа
3. Нажимает "Зарегистрироваться"
4. Система проверяет уникальность email
5. Система создает учетную запись
6. Отправляет email с подтверждением
7. Перенаправляет на страницу входа

**Критерии приемки:**
- Email должен быть уникальным
- Телефон должен быть подстоянным
- Пароль минимум 8 символов
- Подтверждение email обязательно
- Подверждение телефона обязательно
- Первый пользователь становится администратором

---

### UC-02: Вход в систему

**Описание:** Аутентификация пользователя

**Основной поток:**
1. Пользователь вводит email или телфон, и пароль
2. Нажимает "Войти"
3. Система проверяет учетные данные
4. Создает JWT токен
5. Перенаправляет на главную страницу по роли

**Критерии приемки:**
- Сессия 30 минут неактивности
- Блокировка после 6 неудачных попыток
- Запомнить меня на 31 дней

---

### UC-03: Просмотр списка курсов

**Акторы:** Студент, Преподаватель

**Основной поток:**
1. Пользователь заходит в раздел "Курсы"
2. Система показывает список доступных курсов
3. Для каждого курса отображается:
   - Название
   - Название предмета
   - Преподаватель
   - Количество заданий
   - Прогресс (для студентов)
   - Статус (активный/завершенный)

**Критерии приемки:**
- Только курсы текущего семестра
- Цветовое кодирование прогресса
- Сортировка по дате или названию, или преподавателям

---

### UC-04: Просмотр заданий курса

**Акторы:** Студент, Преподаватель

**Основной поток:**
1. Пользователь выбирает курс
2. Система показывает задания курса
3. Группировка заданий:
   - Текущие (дедлайн не наступил)
   - Просроченные
   - Сданные
   - Проверенные

**Критерии приемки:**
- Подсветка ближайших дедлайнов
- Отображение статуса и баллов
- Фильтрация по типу задания

---

### UC-05: Загрузка учебной работы

**Актор:** Студент

**Основной поток:**
1. Студент выбирает задание
2. Нажимает "Загрузить работу"
3. Выбирает файл (PDF, DOC, DOCX, ZIP, RAR)
4. Добавляет комментарий
5. Нажимает "Сдать"
6. Система сохраняет файл
7. Отправляет уведомление преподавателю

**Критерии приемки:**
- Проверка размера файла (до 50 МБ)
- Drag & drop загрузка
- Автосохранение черновика

---

### UC-06: Просмотр оценок и комментариев

**Акторы:** Студент, Преподаватель

**Основной поток:**
1. Пользователь заходит в проверенное задание
2. Система показывает:
   - Итоговый балл
   - Оценки по критериям
   - Комментарии преподавателя
   - Возможность скачать работу с комментариями

**Критерии приемки:**
- Расчет итогового балла с весами
- Цветовая индикация оценки
- Уведомление о новой обратной связи

---

### UC-07: Создание нового курса

**Актор:** Преподаватель

**Основной поток:**
1. Преподаватель нажимает "Создать курс"
2. Заполняет форму:
   - Название курса
   - Код курса
   - Описание
   - Академический год
   - Семестр
   - Критерий оценивания
3. Сохраняет курс
4. Система назначает преподавателя владельцем

**Критерии приемки:**
- Уникальный код курса
- Автогенерация кода
- Возможность клонирования курса

---

### UC-08: Создание задания с критериями

**Актор:** Преподаватель

**Основной поток:**
1. Преподаватель выбирает курс
2. Нажимает "Добавить задание"
3. Заполняет параметры:
   - Название, тип, описание
   - Максимальный балл
   - Критерие оценивания
   - Дедлайны
   - Настройки повторной сдачи
4. Создает рубрику (критерии оценки)
5. Публикует задание

**Критерии приемки:**
- Сумма весов критериев = 100%
- Валидация дедлайнов
- Шаблоны рубрик

---

### UC-09: Проверка студенческой работы

**Актор:** Преподаватель

**Основной поток:**
1. Преподаватель заходит в "Работы на проверку"
2. Выбирает работу
3. Открывается интерфейс проверки
4. Выставляет баллы по критериям
5. Добавляет комментарии
6. Сохраняет/отправляет оценку
7. Система уведомляет студента

**Критерии приемки:**
- Автосохранение черновика
- Валидация баллов
- Расчет итогового балла
- Таймер времени проверки

---

### UC-10: Просмотр аналитики успеваемости

**Акторы:** Преподаватель, Администратор

**Основной поток:**
1. Пользователь заходит в "Аналитика"
2. Система показывает дашборд:
   - Средний балл по курсу
   - Распределение оценок
   - Статистика сдачи работ
   - Статистика присутствие студентов
   - Топ студентов
   - Сравнение групп
3. Возможность экспорта отчетов

**Критерии приемки:**
- Расчет метрик в реальном времени
- Кэширование данных
- Пользовательские отчеты
- Выявление аномалий или ошибок

---

### UC-11: Управление пользователями системы

**Актор:** Администратор

**Основной поток:**
1. Администратор заходит в "Управление пользователями"

- Защита последнего администратора
- Уведомления пользователям

## 5. Бизнес-правила

### 5.1. Правила доступа

| ID | Правило | Описание |
|----|---------|----------|
| BR-01 | Доступ к курсам | Студент видит только свои курсы, преподаватель - свои |
| BR-02 | Доступ к работам | Студент не видит работы других студентов |
| BR-03 | Доступ к оценкам | Преподаватель видит все оценки своего курса |
| BR-04 | Административные права | Только администратор может менять роли пользователей |

### 5.2. Правила оценивания

| ID | Правило | Описание |
|----|---------|----------|
| BR-05 | Расчет оценки | Итоговый балл = Σ(балл × вес) по всем критериям |
| BR-06 | Максимальный балл | Нельзя выставить больше максимального балла задания |
| BR-07 | Изменение оценки | Только преподаватель, выставивший оценку, может ее изменить |
| BR-08 | Округление | Округление до двух знаков после запятой |

### 5.3. Правила сроков

| ID | Правило | Описание |
|----|---------|----------|
| BR-09 | Загрузка после дедлайна | Автоматический статус "просрочено" |
| BR-10 | Проверка после дедлайна | Уведомление администратору |
| BR-11 | Автоматическое закрытие | Закрытие задания после дедлайна проверки |


### 5.4. Правила данных и конфиденциальности:

- BR-12: Хранение работ: Сданные работы хранятся в системе минимум 3 года (согласно академическим стандартам).
- BR-13: Экспорт персональных данных: По запросу пользователя система должна предоставить все его персональные данные и работы в структурированном виде (GDPR/ФЗ-152).
- BR-14: Анонимизация в аналитике: При агрегации данных для дашборда декана/администратора персональные данные студентов могут быть анонимизированы.

### 5.5. Академические правила:

- BR-15: Апелляция/Пересдача: После публикации оценки у студента есть 3 дня на запрос пересмотра (апелляции). Это должно создавать тикет для преподавателя. Количество пересдач ограничено настройками задания (из UC-08).
- BR-16: Итоговая оценка: Формула расчета итоговой оценки по курсу (с учетом весов заданий). Это ключевое правило для аналитики.

## Приложение 1: Глоссарий терминов

**Рубрика (Rubric)** - Система критериев оценки с описанием уровней выполнения  
**Критерий оценки** - Конкретный параметр, по которому оценивается работа  
**Дедлайн (Deadline)** - Крайний срок выполнения задания  
**Вес задания** - Влияние задания на итоговую оценку курса  
**Просроченная работа** - Работа, сданная после дедлайна  
**Статус задания** - Текущее состояние (не начато, в работе, сдано, проверено)  
**JWT токен** - Токен для аутентификации пользователей  
**CSV импорт** - Массовое добавление пользователей через CSV файл  

## Приложение 2: Примеры сценариев использования

**Сценарий 1: Студент сдает лабораторную работу**
1. Иван (студент) входит в систему
2. Переходит в курс "Программирование на Java"
3. Выбирает задание "Лабораторная работа №5"
4. Загружает файл с решением
5. Получает подтверждение сдачи
6. Примерно через 2 дня получает уведомление о проверке
7. Смотрит оценку и комментарии преподавателя

**Сценарий 2: Преподаватель проверяет работы**
1. Мария (преподаватель) входит в систему
